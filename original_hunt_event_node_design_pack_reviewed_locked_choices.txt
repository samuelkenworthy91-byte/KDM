KDM / Lantern Deckbuilder - Reviewed Original Hunt Event Node Design Pack
================================================================================

Revision purpose:
This is a stronger version of the first original hunt event pack. It keeps the dark settlement-survival tone, but now supports two event shapes:

1. Automatic outcome events
   - The event resolves immediately when drawn.
   - Use for weather, road hazards, sudden omens, unavoidable damage, sudden rewards, or mood-setting events where a choice would feel fake.

2. Player choice events
   - The event offers 2 to 4 choices.
   - Some choices may be locked unless the hunting party has the right equipment, weapon type, gear tag, survivor trait, fighting art, innovation, injury/scar/disorder, or Monster Bane knowledge.

Copyright/design note:
The sourceSeed field records which old hunt-event title inspired the slot, but player-facing names, text, and mechanics are original. Do not copy Kingdom Death event names, prose, or rule text into the game.

Repo-specific implementation target:
- Target data file: src/data/events.js
- Existing resolver: src/game/eventLogic.js
- Existing hunt event UI: EventScreen / App event node flow
- Existing gear data: src/data/equipment.js
- Existing survivors and save normalisation: src/game/saveLogic.js

Recommended expanded schema:

Automatic event:
{
  id,
  name,
  description,
  mode: 'automatic',
  autoOutcome: { outcomeText, effects },
  sourceSeed,
  section
}

Choice event:
{
  id,
  name,
  description,
  mode: 'choice',
  choices: [
    { id, text, outcomeText, effects },
    { id, text, lockedUnless, lockedText, outcomeText, effects }
  ],
  sourceSeed,
  section
}

Locked choice requirement shapes to support:
- { type: 'partyHasWeaponType', weaponType: 'bow' }
- { type: 'partyHasGearTag', tag: 'tool' }
- { type: 'partyHasGearId', equipmentId: 'boneBlade' }
- { type: 'partyHasTrait', traitId: 'steady' }
- { type: 'partyHasFightingArt', fightingArtId: '...' }
- { type: 'partyHasFightingArtTag', tag: 'survival' }
- { type: 'partyHasCondition', conditionType: 'scar', conditionId: '...' }
- { type: 'settlementHasInnovation', innovationId: 'monsterStories' }
- { type: 'partyHasMonsterBaneForCurrentQuarry' }
- { type: 'any', requirements: [...] }
- { type: 'all', requirements: [...] }

UI rule:
Show locked choices greyed out with their lockedText. Do not hide them. The player should learn that gear, traits, fighting arts, and innovations open special hunt-event solutions.

Resolver rule:
If mode is automatic, resolve autoOutcome immediately and show the result text. If mode is choice or missing, show choices. Existing events without mode should keep working as normal choice events.

Balance rule:
Locked choices should usually be slightly safer, cleaner, or more tactically useful than the basic choices, but not always pure upside. A locked option can still have a cost if the reward is strong.

SUPPORTED EFFECTS USED BELOW
------------------------------------------------------------------------
These entries mostly use effect keys already present in the current event resolver, such as gainResource, gainRandomResource, gainSettlementMemory, loseHp, healHp, gainSurvival, addPanic, removePanic, addInjury, addScar, addDisorder, nextCombatStartBlock, nextCombatFirstAttackBonus, nextCombatMonsterBonusHp, nextCombatMonsterEnrage, monsterStartsWounded, addQuarryRumour, gainRandomFightingArt, pendingSpecialChildTrait, and chance. If Codex/Gemini introduces any new key, it must add resolver support safely.
